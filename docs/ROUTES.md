# Routes

Status: authoritative for this phase. Defines the Expo Router route table,
guards, tab navigation, modal routes, and deep-link considerations for every
screen in [`MASTER_SPEC.md`](MASTER_SPEC.md) §5. Per-screen UX behaviour is
in [`SCREEN_SPECIFICATIONS.md`](SCREEN_SPECIFICATIONS.md); this document
covers navigation structure only.

## 1. Current state

Two placeholder routes exist today: `src/app/index.tsx` and
`src/app/explore.tsx`, generated by the Expo template. Neither reflects
product navigation. This phase does not create the routes below — it
specifies the target structure for `IMPLEMENTATION_PLAN.md` Phase 1/3.

also a group whose children render inside the bottom tab bar. `workout/` is
a top-level (non-grouped) segment because the active workout is a distinct
modal-like experience that should not be nested inside tab chrome.

## 3. Guards

Four nested guard conditions, evaluated in `_layout.tsx` files via a
redirect-on-mismatch pattern (Expo Router `Redirect`):

1. **Unauthenticated → `(auth)`.** Any route outside `(auth)` redirects to
   `/welcome` if there is no active Supabase session.
2. **Authenticated, onboarding incomplete → `(onboarding)`.** Any route
   outside `(onboarding)` redirects to the first incomplete onboarding step
   if `profiles.onboarding_completed_at` is null. Onboarding steps
   themselves are sequential but a user may navigate backward; forward
   navigation past an incomplete required step is blocked by the step
   screen itself (see `SCREEN_SPECIFICATIONS.md`), not by the router.
3. **Authenticated, onboarding complete → `(tabs)` + `workout/`.** Standard
   application state.
4. **Password-recovery session → `(auth)/reset-password`, exclusively.**
   Opening a password-recovery link establishes a real Supabase session
   (Supabase emits `PASSWORD_RECOVERY`, distinct from `SIGNED_IN`), but it
   is scoped to setting a new password, not to using the app — this guard
   forces `(auth)/reset-password` and nowhere else (not `(tabs)`, not
   `(onboarding)`, not another `(auth)` screen) for the duration of that
   session, overriding rules 1-3 above. Saving the new password signs the
   recovery session out, which hands off to rule 1 (Sign In).

`(auth)`'s own `_layout.tsx` redirects _forward_ to `(tabs)` (or
`(onboarding)` per rule 2) if a session already exists, so a signed-in user
cannot navigate back to Welcome/Sign In.

## 4. Tab navigation

`(tabs)/_layout.tsx` renders the five-tab bar (`MASTER_SPEC.md` §5.1):
Today, Plan, Progress, Coach, Profile. Each tab is its own stack (`today/`,
`plan/`, etc.) so in-tab navigation (e.g. Plan → Workout Detail → Exercise
Detail) preserves independent back-stacks per tab, standard Expo Router tab

- nested stack behaviour.

## 5. Modal routes

Presented with a modal presentation style (`presentation: 'modal'` in the
relevant `Stack.Screen` options), not full navigation pushes:

- Today → Optional Daily Readiness (`today/readiness`)
- Plan → Exercise Swap (`plan/swap/[workoutExerciseId]`)
- Plan → "Why did my programme change?" (`plan/why-changed/[decisionId]`)
- Progress → New BodyScan (`progress/bodyscan/new`) — full-screen modal,
  camera-driven
- Workout → Workout Overview (`workout/[workoutId]/overview`) — modal
  presented before the full-screen Active Workout stack takes over

Everything else is a standard stack push, so back gestures/buttons behave
predictably.

## 6. The Active Workout stack

`workout/[workoutId]/active.tsx` and its child exercise screens form an
independent stack outside `(tabs)`, for two reasons documented in
`ARCHITECTURE.md` §5 and `MASTER_SPEC.md` §13:

- It must be reachable and resumable from a system notification or an app
  relaunch mid-workout, independent of which tab was last active.
- Its state (current set, rest timer, offline queue) lives in the
  client-side domain layer (`src/domain/workout`), not in route params, so
  backgrounding/foregrounding the app or a network drop never loses
  progress. The route only reflects _which_ workout/exercise is being
  viewed; it is not the source of truth for workout state.

Leaving the Active Workout stack (e.g. switching tabs) does not end the
workout — a persistent "resume workout" affordance is shown across tabs
while a workout is `in_progress` (exact placement:
`SCREEN_SPECIFICATIONS.md`).

## 7. Deep links

`app.json` already declares `"scheme": "murphymethod"`. Deep-link targets
supported from MVP notification payloads (`MASTER_SPEC.md` §28):

| Notification type                                                  | Deep link target                                                                 |
| ------------------------------------------------------------------ | -------------------------------------------------------------------------------- |
| `workout_reminder`                                                 | `workout/[workoutId]/overview`                                                   |
| `missed_start`                                                     | `workout/[workoutId]/overview` (with Quick/Reschedule/Skip surfaced immediately) |
| `progress` (PR)                                                    | `(tabs)/progress/records`                                                        |
| `bodyscan_due`                                                     | `(tabs)/progress/bodyscan`                                                       |
| _(Supabase Auth, not an MVP notification)_ password recovery email | `(auth)/reset-password` — see §3 rule 4                                          |

A deep link into a guarded route re-runs the guard chain in §3 first (e.g. a
notification opened while signed out lands on Sign In, then continues to the
original target after authentication — the target is preserved as a pending
redirect, not dropped).

## 8. Web

`app.json` already configures static web output. No route in this table is
web-incompatible in principle, but the Active Workout stack's camera-adjacent
and gesture-heavy UX (§6) and the BodyScan capture modal are treated as
mobile-primary; their web behaviour is explicitly out of scope for MVP (see
[`DEFERRED.md`](DEFERRED.md)) and this phase does not design a web-specific
fallback for them.
